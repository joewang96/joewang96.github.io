Lose the Graphics: Building a Simple, Function-First Weather App

Position:

N/A, personal project

Dates:

October 2017

Tags:

Vue.js, Flask, web app, UI design

The reasoning behind the project

Weather applications are a very common theme for development projects however most share a similar design, opting for intense graphics or full background colors to symbolize the current state of the weather. From self experience I have had issues trying to find myself around weather sites or just find the experience cluttered with data that isn't relevant to what I want to know at the time. The solution - create a weather app tailored for a user that wants to get the current weather data and be able to understand it quickly.

Creating a clean interface

The main purpose of this project was to parse down all available information regarding the weather into what is most relevant for the average user. After talking to a few people, I came up with a set of primary features to display based on common patterns. The app ended up showing the current, daily high, and daily low temperature as well as current humidity, wind speed, and any weather features (such as rain, hail, fog, etc.).

With the content in mind I went about creating the layout that would be both functional and aesthetically pleasing. I ended up going with a muted off-white background accompanied with large user controls showing the clear actions. Lastly, I formatted the data being displayed in terms of heirarchy, based on what would be most pertinent to a user wanting to find quick information. I strucutred the information in a word-based format, allowing the user to quickly read the text at a glance and get the most pertinent information.

Final landing page design of the web app. Showcases the two options of either using current location or entering a zip code.



The resulting view showing the data for the location being used. Clean language based interface, with colored text denoting importance and feel (cold vs. warm tones denoting temperature)

Development and deploying to Heroku

The final step in the development processes was getting the site accessible to the public. Initially I was thinking about coding the site as a static webpage allowing it to be deployed through Github pages, however I believed that this would limit the fluid experience by requiring the user to give browser permissions in order to use the geolocation features.

Additionally, I wanted to gain the experience of deploying an actual web app on my own for a project. I decided to use Flask as the application framework due to prior experience, and Waitress to help manage server performance since multiple users could be using the app at the same time. After a few sessions of debugging my Procfiles and server side code, I finally managed to deploy to Heroku.
